Date : 04-Jul-2026

When gallery-dl processes an entire board at once, it fetches board-level metadata from Pinterest, which unfortunately omits the original creator's attribution (origin_pinner or via_pinner).
In these cases, it defaults the attribution to the person who saved the pin (which ends up being your account, the board owner). 
However, when you download an individual pin URL directly, it hits the pin endpoint which contains the complete original metadata.

To fix this comprehensively, modified src/parser.py.

What was changed:
Board Expansion via gallery-dl: When parse_links() reads links, it now uses gallery-dl (with the -N flag) behind the scenes to "expand" any board or profile URL into a flat list of individual pin URLs.

Full Metadata: Because the script now feeds these direct pin URLs into the downloader loop, every pin is processed individually. 
               This ensures user get the full, original metadata for every file in the board.

Accurate Rate-Limiting: As a huge bonus, expanding boards into individual pins means SESSION_SIZE config (50) and new delay times correctly apply to each individual pin within a board. 
                        Previously, downloading a board with 1,000 pins would have bypassed limits and downloaded all 1,000 at once!

Added an indicator Extracting individual pin URLs from boards/profiles... that user will see in the terminal, as expanding a large board URL might take a few moments before the download session begins.

Date : 06-Jul-2026

- Resolved Major Issue #5.1 [Fix: Pin Duplication during Board Downloads].

- Because gallery-dl outputs the pin URL once for each media item inside a pin when extracting a board. 
    This meant that a pin with 3 media items would result in 3 identical pin URLs being sent to the downloader
    which then downloaded all 3 media items for each of those 3 identical links (resulting in 9 downloads).

- Updated parser.py to ensure that duplicate URLs are filtered out:

- When extracting URLs from a board, it now immediately deduplicates the list of pin URLs 
    while preserving their original order.
- It also globally deduplicates all processed URLs before handing them off to the downloader, 
    which ensures that if multiple boards share the same pin, it will still only download once.


- Resolved Major Issue #5.2 [Fix: Log Duplication multiple media items].

- When organize_metadata() gathers the downloaded metadata files, 
    it will check if an identical metadata entry (with the same pin_id and file_type) already exists in the log list for that pin. 
    If it does, it won't append it again.

- This means that if a single pin link contains 8 images 
    Log.json will now only contain one clean metadata entry for that link instead of repeating it 8 times.

Date : 08-Jul-2026

- Added retry logic for failed links.
    - Added MAX_RETRIES = 3 to config.py. The script will now attempt to download failed links up to a maximum of 3 times per session (1 initial attempt + 2 retries).

- Main Loop Orchestration
    - Rewrote the main loop in main.py. The script now completes a full "Pass 1" of all links.
    - If any links fail, it automatically starts a "Pass 2" containing only the failed links.
    - It continues this process until all links succeed or MAX_RETRIES is reached.

- Dashboard UI
    - Updated the Rich dashboard to display the current pass (e.g., Downloading (Pass 2/3): url...).
    - When a link fails on Pass 1, it marks a failure. If it subsequently succeeds on Pass 2, the UI automatically reverses that failure count and adds it to the success count, keeping the progress bar perfectly accurate.

- Smart Logging
    - Updated logger.py to prevent duplicate log entries for the same link.
    - Instead of adding a new entry every time a link is retried, the logger now finds the existing entry for that URL and updates it with the final status and total duration.
    - Added an attempt_count field to Log.json so you can see exactly how many times a link had to be retried before it succeeded or ultimately failed.

- Exact behavior
    - Pass 1 (The Main Run): The script reads every link in Links.txt file (let's say 120 links) and starts downloading them sequentially.
    - Cooldown Sessions: As it works through those 120 links, it will still take its long break (the MIN_COOLDOWN / SESSION_SIZE) every 50 links, just like before.
    - End of Pass 1: Once it finishes attempting all 120 links, Pass 1 is completely finished.
    - Pass 2 (Retry 1): Only after all 120 links are done, it gathers any links that failed (let's say 5 of them failed). It immediately starts Pass 2 and tries to download just those 5 links.
    - Pass 3 (Retry 2): If 2 of those 5 links fail again, it will start Pass 3 and try those 2 links one last time.
